Abstract Accessibility guidelines [1]-[3] emphasize the need to
allow the user to interact with web pages not only through a
pointing device, but through the keyboard as well. The
specification of HTML4 [4] came with the access keys but they
have resulted in an unexpected failure [5]. This paper proposes
a new system which deters conflicts between web page access
keys and keyboard shortcuts used by various OS applications.
Keywords- Web accessibility, HTML Access keys, keyboard
shortcuts, user agent, assistive technology, w3c guidelines
I. I
NTRODUCTION
To overcome HTML access keys limitation, we propose a
new system for web navigation with the aim to:
Avoid conflict between the access keys of a web page
on one side and keyboard shortcuts of user agents and
assistive technologies on the other side.
Facilitate the use of access keys by abandoning the
tradition of the simultaneous use of several keys to
activate an access key.
Provide a unique activation mode independent of any
user agent and operating system.

To validate our solution, we implemented as part of a web
navigation system trough keyboard (Fig 1).
A. The Parser of Web interfaces
Our parser performs the following functions:
Create a HTML DOM object from the URL of a web
interface.
Traverse the DOM and find all nodes corresponding
to link elements (Tag A having an attribute HREF) or
form controls.
Produce a data structure that associates each value of
access key to its corresponding element. This data
structure will be used by the engine of access keys
rendering as well as by the engine of access keys
activation.
Because a keyboard access should not lead to confusion,
we decide to assign the same access key to each identical link.
In fact, criterion 6.1-6.5 of the RGAA French guideline [6]
asked to verify for each web page, if each identical link has
the same purpose and target (Priority 1).
B. The Engine of Access Keys Rendering
The engine traverses the DOM tree and modifies HTML
elements to enable access keys rendering. It appends the
access key value of a hyperlink element to its inner text. This
is an adaptive link annotation [8]. For a control of a form the
value of its access key is rendered as a part of its label. The
engine renders for each control of a form its access key even if
it has originally no label.
The proposed solution allows users as well as assistive
technologies to identify the values of access keys and
distinguish them from the rest of the page content. Access
keys values are integrated in a span element having the
attribute title set to the value "accesskey". By pointing that
span element, a user can see a tooltip indicating that it is an
access key. The element span is preceded by the image of a
key having its alt attribute set to the value accesskey.
Therefore, the screen readers recognize the word accesskey
before reading its corresponding digits. In addition, each
screen reader automatically translates the word "accesskey in
the language specified in its configuration (TABLE I).

C. The Access Keys Listener
This is a keyboard event listener. It listens the entered
keystrokes of numeric keys. It concatenates the listened digits
to form the number corresponding to the access key. At the
end of the listening period, the access key value will be
transferred to the engine of access keys activation. Our access
keys listener works as similar as a remote control that sends
the number of the requested channel to the receiver.
D. The engine of Access Keys Activation
Upon receiving an access key, the engine of access keys
activation will give the focus to the convenient element.
Giving the focus to an element differs from one element to
another according to its type.
The focus allows to follow up a link while it puts the
cursor in a text field or a text area to prepare it to receive
user's keystrokes. The focus makes a select button selected
and a check box checked. Giving focus to a submit button
leads to the sending of form data. However, giving focus to a
reset button causes the resetting of the form controls.
If no element corresponds to the received access key, the
engine performs no action and immediately gives a hand to
the listener of access keys.
III. C
ONCLUSION AND
F
URTHER
W
ORK
The proposed solution has offered to web apps their own
access keys listener independent from the keyboard shortcuts
listener of the user agent. Thus, it has successfully avoided
conflicts and made the activation of access keys easier:
It has provided a unified method to activate access
keys regardless of the used OS and user agent.
Visually impaired and persons with Alzheimer's no
longer have to guess the convenient keystrokes to
their work environments.
The keystrokes of an access key have to be entered
successively. Activating an access key no longer
requests to enter simultaneously two up to four keys,
which requires a large gym from a disabled person
[5].
It allowed direct use of the keypad that is still ignored
by almost all desktop user agents. The numeric keys
that make up our access keys are the most accessible
on mobile devices.
Since personalization can improve accessibility as well as
performance/user interface [7]-[8], w3c defined the IndieUI
user context [9]. It provides relevant information about users
settings and preferences (color, type/font, media and screen
reader) which can be accessed help to a user settings listener.
In our further work we will study how personalization can
approve the performance of our access keys listener as well as
the rendering of our access keys.
